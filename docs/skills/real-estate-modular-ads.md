# Real Estate Modular Ads

Skill local para investigar una propiedad o proyecto y convertir esa información en un sistema modular de grabación y testing para Meta Ads.

## Qué resuelve

Este skill sirve cuando no basta con escribir un copy o revisar un video ya grabado. Está pensado para la etapa previa:

- investigar una propiedad o proyecto online
- identificar ventajas, desventajas y claims utilizables
- traducir esa investigación a hooks, ángulos y pruebas visuales
- preparar el paquete operativo para grabar y testear anuncios modulares

## Entregables esperados

La salida del skill debe incluir:

1. `Research brief`
   hechos verificados, oferta, audiencia probable, objeciones, claims permitidos y claims no confirmados
2. `Ventajas y desventajas`
   útiles para ventas y para filtrar expectativas
3. `Hooks, ángulos, filtros suaves y CTA`
   como bloques separados y reutilizables para grabar por tandas
4. `Plan de grabación`
   hooks, ángulos, shot list, b-roll, checklist de rodaje y CTA
5. `Preparación del ad`
   on-screen text, orden recomendado de módulos y matriz de testing

## Qué tipo de fuentes prioriza

Primero:

- sitio oficial del proyecto o builder
- listing oficial o MLS si está disponible
- landing pages activas de campaña

Después:

- Google Maps y contexto geográfico
- portales inmobiliarios
- fuentes públicas para contexto de zona cuando aporten algo útil

Si un dato es inestable, el skill debe tratarlo como condicional o pedir validación antes de convertirlo en claim.

## Qué no debería hacer

- inventar incentivos, financiamiento o cifras no verificadas
- presentar especulación como hecho
- construir hooks solo “bonitos” pero desconectados de la propiedad real
- recomendar ángulos imposibles de demostrar en cámara
- dejar los ángulos en abstracto sin decir qué frase o idea concreta debe comunicar el speaker
- meter el nombre del proyecto, comunidad o modelo dentro del copy final si no hace falta para vender
- usar el nombre del proyecto como palabra clave de WhatsApp, CTA o texto en pantalla salvo que se pida explícitamente
- meter por defecto un bloque de `rodaje por escenas` si el usuario no lo pidió

## Regla de naming en piezas

Por defecto, el skill debe tratar el nombre del proyecto, comunidad o modelo como dato interno de investigación, no como parte obligatoria del anuncio.

Aplicarlo así:

- mantener el nombre en el `research brief` y en la identificación interna del caso
- quitarlo de hooks, CTAs, palabras clave, on-screen text y variantes de copy
- reemplazarlo por lenguaje más útil para conversión, por ejemplo `esta casa`, `esta opción`, `esta propiedad` o una keyword genérica como `INFO` o `CASA`

Solo incluir el nombre del proyecto en la pieza final cuando el usuario lo pida de forma explícita o cuando sea estratégicamente necesario para búsqueda de marca.

## Flujo resumido

1. Recibir la propiedad, proyecto o enlace base.
2. Investigar fuentes primarias y complementar con contexto de zona.
3. Separar hechos confirmados de inferencias.
4. Detectar ventajas, limitaciones y riesgos de mensaje.
5. Diseñar hooks, ángulos concretos, filtros suaves y CTA por separado.
6. Añadir filtro suave de precalificación cuando la pieza lo necesite.
7. Preparar shooting sheet y matriz de testing.

## Regla de precalificación suave

Cuando el skill genere guiones o variantes listas para grabar, debe poder añadir una capa ligera de filtrado comercial sin convertir el anuncio en un formulario.

Usarlo con lenguaje como:

- `si tienes crédito alrededor de 620 o más`
- `si tienes ingresos estables comprobables`
- `si cuentas con aproximadamente $20,000 para el down payment`

Reglas:

- no prometer aprobación
- no usar porcentajes solos si se pueden convertir a dólares aproximados
- si el cálculo depende del precio actual, usar una cifra estimada y marcarla como aproximada
- si no hay base suficiente para calcular, dejarlo como dato a validar

## Regla de modularidad

Por defecto, el skill no debe devolver anuncios armados como bloque largo.

Debe priorizar:

- lista de hooks
- lista de ángulos
- lista de filtros suaves de precalificación
- lista de CTA

La idea es que el usuario pueda grabar primero todos los hooks, luego todos los ángulos, luego todos los filtros suaves y al final todos los CTA para recombinar después en edición.

Los `soft qualifiers` no deben meterse todos juntos por defecto. La lógica correcta es:

- grabar varias opciones
- usar `1` qualifier en la mayoría de piezas
- usar `2` solo cuando se quiera filtrar más fuerte
- evitar que el anuncio se vuelva una lista de requisitos

## Criterios heredados de guiones de ads

Además de la modularidad, el skill debe respetar estas reglas base al generar piezas:

- cada anuncio armado después debe sostener `un solo ángulo principal`
- el `hook` debe funcionar en los primeros `1-2` segundos
- el `CTA` debe decir qué hacer y qué recibirá la persona después
- si se mencionan ayudas, incentivos, entrada o aprobación, usar lenguaje condicionado como `si calificas` o `según tu perfil`
- no mezclar demasiadas promesas en el mismo bloque de ángulo
